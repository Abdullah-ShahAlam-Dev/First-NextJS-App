Yeh "Tameez Se" Push Kyun Nahi Ho Raha?
1. Main Repo vs Subtree Repo ka Masla:
Jab aapne git pull wali command chalayi, toh aapki Main Repo (SMIT-BATCH-17) ko pata chal gaya ke GitHub se README aa gaya hai. Main repo khush ho gayi.

2. Subtree Ki Apni Alag Duniya:
Lekin, jab aap git subtree push chalate hain, toh Git kya karta hai? Wo aapke PortableGit folder ko uthata hai aur background mein uski ek bilkul nayi aur alag history (naye Commit IDs) banata hai taake usko bheja ja sake.

3. Hashes (IDs) Ka Mismatch:
GitHub par jo akela README ka "Initial Commit" para hai, uska ID (jaise a88dcf0...) aur jo Git Subtree ne naya ID banaya hai, wo aapas mein match nahi karte!
GitHub dekhta hai aur bolta hai: "Tumhari main repo mein toh README hai, lekin jo chota dabba (subtree) tum mujhe bhej rahe ho, uski history mere pehle commit se match nahi kar rahi. Yeh toh koi doosri duniya ka code lag raha hai." Isliye wo "non-fast-forward" keh kar push reject kar deta hai.

Khulasa (Summary)
Subtree hamesha pehli dafa nakhre karta hai agar GitHub par repo banate waqt aapne "Add README" par tick kar diya ho. Agar GitHub par repo bilkul khaali (empty) hoti, toh yeh masla kabhi nahi aata aur direct "tameez se" push ho jata.

Kyunke ab Hashes match nahi kar rahe, toh is block ko torrne ka wahid rasta yahi hai ke hum ek dafa Force Push (Hathora) chala kar GitHub ki us purani history ko overwrite kar dein.


Sirf ek dafa yeh force push lagega, uske baad GitHub aur aapka PC ek page par aa jayenge aur sari zindagi aapki .sh script bilkul "tameez se" normal push karegi!
